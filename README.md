# 音立方 BLE 音频流 Demo

## 一、项目概述

### 背景
音立方是一款蓝牙录音设备（外设），支持持续录音并通过 BLE 实时将音频流传输到手机。本项目是一个 Android Demo，验证手机与设备之间的蓝牙通信能力。

### 核心目标
1. **BLE 连接**：扫描、发现、连接音立方设备
2. **实时音频流**：接收设备推送的 MP3 音频流并展示
3. **录音控制**：远程控制设备开始/停止录音
4. **状态展示**：显示连接状态、电量、录音状态等

## 二、技术要点

### 通信协议
- **通信方式**：BLE（低功耗蓝牙）
- **指令格式**：APP 发送 `YWT_BLE&...`，设备回复 `YWT_DEV&...`
- **音频格式**：MP3 流式传输
- **服务 UUID**：`001120a0-2233-4455-6677-88995a5b5c5d`

### 核心 UUID
| 功能 | UUID |
|------|------|
| 蓝牙主服务 | `001120a0-2233-4455-6677-88995a5b5c5d` |
| 音频 Notify | `001120a1-2233-4455-6677-88995a5b5c5d` |
| 指令 Notify | `001120a3-2233-4455-6677-88995a5b5c5d` |
| 写入 Write | `001120a2-2233-4455-6677-88995a5b5c5d` |

### BLE 传输挑战
| 问题 | 说明 |
|------|------|
| 带宽有限 | BLE 4.2 理论最大 ~1Mbps，实际有效载荷约 2-5 KB/s |
| 包大小限制 | 单次 Notify 最大 20 字节（协商前），协商后最大 244 字节 |
| 连接间隔 | 最小 7.5ms，实际传输频率受限 |
| 易受干扰 | 2.4GHz 频段拥挤，WiFi、微波炉都会干扰 |

## 三、架构设计

### 整体分层（三层）

```
┌─────────────────────────────────────────────────┐
│                   UI 层                          │
│  MainActivity / Fragment                        │
│  观察状态，展示数据，响应用户操作                    │
└──────────────────┬──────────────────────────────┘
                   │ 观察 LiveData / Flow
┌──────────────────▼──────────────────────────────┐
│               业务层 (Repository)                │
│  DeviceRepository                               │
│  负责：指令组装、协议解析、音频缓冲管理              │
└──────────────────┬──────────────────────────────┘
                   │ 调用
┌──────────────────▼──────────────────────────────┐
│               蓝牙层 (BleManager)                │
│  BleManager                                     │
│  负责：扫描、连接、MTU协商、数据收发                │
│  纯蓝牙操作，不关心业务协议                         │
└─────────────────────────────────────────────────┘
```

### 核心原则
- **BleManager 不知道协议内容**：只负责原始字节的收发
- **Repository 不关心蓝牙细节**：只通过 BleManager 接口操作
- **UI 只观察状态**：不直接操作蓝牙

### 目录结构

```
com.example.bleaudio/
├── MainActivity.kt              // UI 层
│
├── ble/                         // 蓝牙层（可复用）
│   ├── BleManager.kt           // 蓝牙操作管理
│   └── ConnectionState.kt      // 连接状态枚举
│
├── repository/                  // 业务层
│   ├── DeviceRepository.kt     // 设备业务逻辑
│   └── DeviceState.kt          // 设备状态数据类
│
└── protocol/                    // 协议定义（可复用）
    └── YinLiFangProtocol.kt   // UUID、指令常量
```

### 数据流向

```
设备 Notify
    │
    ▼
BleManager.receivedData (原始字节)
    │
    ▼
DeviceRepository.parseResponse()
    ├── 音频数据 → audioStream → UI 播放/展示
    └── 指令响应 → commandResponse → 更新 deviceState → UI 展示

用户点击按钮
    │
    ▼
DeviceRepository.startRecording()
    │
    ▼
BleManager.writeData(原始字节)
    │
    ▼
设备
```

## 四、稳定性设计

### 1. 连接层 — 保活与重连
- **心跳检测**：定期发送轻量指令确认连接存活
- **自动重连**：断开后指数退避重连（1s → 2s → 4s → 8s，上限 30s）
- **MTU 协商**：连接建立后立即请求 244 字节 MTU

### 2. 数据层 — 缓冲与容错
- **环形缓冲区**：设置 3-5 秒的音频缓冲，平滑突发延迟
- **序列号检测**：在数据流中加入序号，检测丢包和乱序
- **静音填充**：丢包时用静音帧填充，避免播放卡顿

### 3. 传输层 — 流控与优先级
- **指令队列**：非实时指令排队发送，避免阻塞音频流
- **背压机制**：如果缓冲区快满了，暂停发送控制指令

### 4. 应用层 — 状态机管理
```
未连接 → 连接中 → 配对中 → 已就绪 → 传输中
   ↑                              │
   └──────── 自动重连 ─────────────┘
```

### 5. 异常处理
| 异常场景 | 处理方式 |
|---------|---------|
| BLE 信号弱 | 降低音频码率 / 提示用户靠近设备 |
| 缓冲区溢出 | 丢弃最旧数据，保持实时性 |
| 设备断电 | 保存当前状态，提示用户 |
| 协议解析错误 | 跳过当前帧，等待下一帧 |
| 后台被杀 | 前台 Service 保活 + 通知栏常驻 |

## 五、复用说明

| 模块 | 复用场景 |
|------|---------|
| `BleManager` | 任何 BLE 项目都能直接用，换 UUID 即可 |
| `YinLiFangProtocol` | 同系列设备的协议常量集中管理 |
| `DeviceRepository` | 如果换蓝牙库，只改 BleManager，Repository 不用动 |
